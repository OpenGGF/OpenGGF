Audio Rewind Deferred Design Orchestration
==========================================

Requirements
------------

Goal: turn the deferred audio rewind design items in
`AUDIO_REWIND_BLUEPRINT.tmp.txt` into implementable architecture and task
plans without changing the already-green restore-forward Tier 0 through Tier 3
behavior.

Design scope:
- Deterministic frame-owned audio core boundary.
- Explicit pause and frame-step audio semantics.
- Engine-owned output FIFO and OpenAL drain boundary.
- PCM history ring for audible reverse scrub presentation.
- Rewind release crossfade and tape-coast interaction.
- Benchmark and memory/performance budget design.
- Optional cycle-aware scheduler design, gated behind future parity evidence.

Non-goals:
- Do not make OpenAL source ids, buffer ids, or queued native buffers
  authoritative rewind state.
- Do not replace the existing restore-forward snapshot path.
- Do not make cycle-level scheduling a prerequisite for current Tier 3
  sample parity.
- Do not implement feature code in this design artifact.

Acceptance criteria:
- Every deferred item has an owner boundary, data contract, lifecycle, tests,
  and rollout order.
- The plan preserves existing focused-green audio rewind tests.
- Presentation-only state remains isolated from authoritative audio state.
- Memory and performance budgets are explicit enough to become test/benchmark
  gates.
- Cycle-aware scheduling is clearly optional and separately gateable.

Assumptions:
- The current snapshot/restore-forward implementation remains the stable base.
- Gameplay rewind remains frame-indexed; any sub-frame or cycle model must sit
  inside audio runtime internals and not change gameplay snapshot identity.
- Audible reverse audio is UX presentation, not the source of truth for future
  playback.

Risks:
- Moving audio advancement out of OpenAL refill can change timing and expose
  latent assumptions in `LWJGLAudioBackend.fillBuffer(...)`.
- PCM history can grow quickly if configured by seconds, sample rate, and
  stereo width without hard limits.
- Frame-step policy can confuse trace/debug behavior if audible output and
  authoritative state are coupled.
- Cycle-aware scheduling can balloon scope if treated as mandatory before
  evidence shows frame-level parity is insufficient.

Exploration Synthesis
---------------------

Local evidence:
- `AudioManager.advancePausedFrameStepAudio()` currently delegates to
  `update()` and is explicitly marked as the Tier 0 frame-step seam.
- `LWJGLAudioBackend.fillBuffer(...)` currently advances `currentStream` and
  `sfxStream` only when OpenAL needs buffers, so authoritative SMPS/chip time
  is coupled to presentation refill cadence.
- `SmpsDriver.read(short[])` and `SmpsSequencer.read(short[])` are the current
  sample-production boundary.
- `AudioBenchmarkMemoryProbe` and `AudioRegressionTest` already provide audio
  render timing and allocation measurement patterns.
- `RewindBenchmark` already writes JSON phase timing to
  `target/rewind-benchmark-results.json`, making it a good target for audio
  rewind benchmark extension.

Agent exploration:
- Deterministic core explorer: current authoritative audio phase still advances
  from `LWJGLAudioBackend.fillBuffer(...)` through `SmpsDriver.read(...)`, so
  OpenAL processed-buffer cadence decides SMPS/chip phase. At 48 kHz,
  `STREAM_BUFFER_SIZE = 1024` is about 21.3 ms, while a 60 Hz gameplay frame is
  about 16.67 ms. This confirms frame-owned audio needs an explicit
  `AudioFrameClock` and a backend drain that no longer calls `SmpsDriver.read`.
- PCM presentation explorer: the best PCM history insertion point is after
  final music/SFX mixing in `LWJGLAudioBackend.fillBuffer(...)`, immediately
  before `alBufferData`. Reverse scrub must read only this history and must not
  advance SMPS streams. Sample-rate changes and backend reinit invalidate the
  ring.
- Benchmark explorer: existing `RewindBenchmark` and `AudioRegressionTest`
  provide timing/allocation patterns, but audio rewind lacks dedicated
  measurements for `AudioManager.captureLogicalSnapshot()`,
  `restoreLogicalSnapshot()`, `AudioKeyframeStore.replayTo(...)`, command
  timeline growth, SMPS/chip retained size, PCM history memory, and
  paused-frame-step produced samples.

Recommendation:
- Introduce a deterministic audio runtime layer above `SmpsDriver` and below
  `AudioManager`, then make `LWJGLAudioBackend` drain PCM from that runtime
  instead of owning advancement.
- Normalize frame command ordering before runtime migration: commands emitted
  during gameplay frame `N` should affect audio frame `N` under a documented
  begin-run-advance-end contract.
- Add PCM history only after the runtime owns an output FIFO; keep it
  disposable and bounded.
- Keep cycle-aware scheduling as a design extension of the deterministic
  runtime, not as the first implementation step.

Architecture Decision
---------------------

Ownership:
- `AudioManager` remains the command-recording and high-level intent boundary.
- New `DeterministicAudioRuntime` owns authoritative audio advancement,
  command consumption, SMPS driver state, chip snapshots, output FIFO, and
  frame/sample accumulators.
- `LWJGLAudioBackend` becomes a presentation drain for already-produced PCM and
  still owns OpenAL device/source/buffer objects.
- New `PcmHistoryRing` owns reverse-scrub presentation history only.
- New benchmark helpers live under test/benchmark code and read runtime
  counters without mutating production behavior.

Lifecycle:
- Gameplay session start creates a fresh audio runtime after audio profile and
  donor loaders are known.
- Hard rewind boundaries reset runtime, command timeline, FIFO, and PCM
  history.
- Rewind seek restores an audio keyframe into the runtime, replays commands and
  deterministic time to the target frame, then flushes/reprimes presentation.
- Device reinitialization only rebuilds OpenAL presentation queues; it does not
  mutate authoritative audio state.

Data flow:
1. Gameplay calls `AudioManager`.
2. `AudioManager` records `AudioCommand` with frame/order and sends it to the
   deterministic runtime.
3. The runtime consumes commands for a frame, advances SMPS/chip state by a
   fixed frame slice, and appends PCM to its output FIFO.
4. `LWJGLAudioBackend` drains FIFO into OpenAL buffers without advancing
   authoritative time.
5. Optional `PcmHistoryRing` mirrors produced PCM for reverse presentation.

Rollback:
- Keep existing `AudioBackend` playback path behind a compatibility adapter
  until deterministic runtime parity tests pass.
- Runtime can be disabled by configuration or test profile, returning to the
  current backend-driven stream model.

Feature Design
--------------

Deterministic runtime APIs:
- `submit(AudioCommand command, long frame, int order)`
- `advanceFrame(long frame, FrameAudioMode mode)`
- `captureSnapshot()`
- `restoreSnapshot(AudioRuntimeSnapshot snapshot)`
- `replayTo(long frame, AudioCommandTimeline timeline)`
- `drainPcm(short[] target, int frames)`
- `flushPresentationFifo()`

Frame audio modes:
- `NORMAL`: advance authoritative state and make produced PCM available.
- `SILENT_STEP`: advance authoritative state but do not send PCM to live
  presentation.
- `INTERNAL_REPLAY`: advance for seek/replay with no presentation side effects.
- `PRESENTATION_ONLY_REVERSE`: do not advance authoritative state; read PCM
  history for audible scrub.

Frame contract:
1. `beginGameplayAudioFrame(N)` sets command timeline frame/order state.
2. Gameplay frame `N` runs and emits audio commands.
3. `advanceAuthoritativeFrame(N, mode)` consumes frame `N` commands and renders
   exactly one gameplay-frame worth of audio samples.
4. Produced PCM is enqueued to presentation unless mode suppresses it.
5. `endGameplayAudioFrame(N)` records counters and clears transient frame
   state.

`AudioFrameClock`:
- Uses rational/fixed-point accumulation for `sampleRate / gameFrameRate`.
- Stores sample-rate, region/frame-rate identity, numerator, denominator, and
  remainder in runtime snapshots.
- Produces exact totals over long runs even for non-divisible rates.

Output FIFO:
- Stores interleaved stereo PCM at runtime output rate.
- Has bounded capacity in frames and reports underrun/overrun counters.
- Is not captured in authoritative keyframes except for optional presentation
  prefill metadata.

PCM history:
- Ring stores recent interleaved stereo PCM frames.
- Capacity configured in seconds and capped by absolute bytes. Proposed
  default: 10 seconds. Proposed hard cap: 60 seconds.
- At 44.1 kHz stereo 16-bit, memory is 176,400 bytes/second, about 1.68 MiB
  for 10 seconds and about 10.1 MiB for 60 seconds.
- Reverse playback reads from a separate cursor and applies a short windowed
  crossfade on entry/exit. Proposed release crossfade: 30-80 ms.
- History is cleared at hard boundaries and after backend/device reset.

Pause/frame-step:
- Paused: no authoritative audio advance.
- Frame-step silent default: advance exactly one gameplay-frame audio slice,
  record counters, suppress live output.
- Optional audible pulse: enqueue only a short crossfaded slice after the
  deterministic step, never by advancing OpenAL-owned streams.

Cycle-aware scheduler:
- Optional module under the deterministic runtime.
- Stores command/chip-write events as fixed-point offsets within a game frame.
- Enabled only for targeted parity tests that prove frame-order timing is not
  sufficient.

Benchmark design:
- Track capture time, restore time, replay-to-frame time, per-frame advance
  time, FIFO drain time, PCM history memory, snapshot retained bytes, command
  count, FIFO underruns/overruns, and allocation deltas.
- Emit JSON alongside existing rewind benchmark output.
- Add budget checks as opt-in benchmark assertions first, then promote stable
  thresholds to CI-friendly tests.

Initial benchmark budgets:
- Command recording overhead: p95 <= 25 us per audio command, p99 <= 75 us.
- Command timeline retained bytes: <= 4 MiB for 10 minutes under a
  ring-loss/results-tally stress workload.
- Logical audio snapshot: mean capture <= 100 us, restore <= 150 us, retained
  size <= 8 KiB.
- Full audio keyframe with SMPS + synth state: mean capture <= 750 us, restore
  <= 1 ms, retained size target <= 32 KiB, hard gate <= 64 KiB.
- Replay from nearest 60-frame keyframe: mean <= 2 ms, p95 <= 5 ms, max <=
  10 ms.
- Forward playback with audio rewind enabled: mean frame overhead <= 0.25 ms,
  p95 <= 0.75 ms.
- Steady-state benchmark after warmup: zero GC collections; allocation rate is
  tracked and budgeted after the first baseline run.

Implementation Plan
-------------------

Task A: deterministic runtime boundary.
- Tests first: fake command timeline drives runtime, runtime advances exactly
  one frame slice, OpenAL drain does not advance SMPS.
- Files: new audio runtime package plus adapter seams in `AudioManager` and
  `AudioBackend`.
- Verification: focused audio runtime tests and existing 74-test rewind suite.

Task B: output FIFO and backend drain.
- Tests first: fill/drain/underrun/overrun behavior, backend drain does not
  call `SmpsDriver.read(...)` directly.
- Files: runtime FIFO classes and `LWJGLAudioBackend` drain path.
- Verification: backend snapshot tests plus audio regression tests.

Task C: pause and frame-step semantics.
- Tests first: paused frame-step advances one frame of authoritative audio in
  silent mode; normal pause advances none.
- Files: `AudioManager.advancePausedFrameStepAudio`, `GameLoop` integration,
  runtime frame modes.
- Verification: frame-step unit tests and trace/rewind focused tests.

Task D: PCM history and reverse presentation.
- Tests first: bounded ring behavior, reverse cursor mapping, crossfade entry
  and release, hard-boundary clearing.
- Files: `PcmHistoryRing`, presentation policy classes, live rewind manager
  hooks.
- Verification: pure PCM unit tests and a fake backend presentation test.

Task E: benchmark/budget extensions.
- Tests first: benchmark JSON includes audio phases and counters when enabled.
- Files: `RewindBenchmark`, `AudioBenchmarkMemoryProbe` helpers, budget docs.
- Verification: opt-in benchmark command plus non-negative/counter tests.
- Proposed command:
  `mvn -Dtest=com.openggf.audio.AudioRewindBenchmark -Dopenggf.audio.rewind.benchmark.run=true test`

Task F: optional cycle-aware scheduler.
- Tests first: frame-level and cycle-level schedules produce different ordered
  chip writes in a synthetic case; disabled mode preserves current behavior.
- Files: audio runtime scheduler internals only.
- Verification: synthetic timing tests; enable in parity tests only after a
  demonstrated need.

Forward playback quality regression follow-up:
- User report: fades were not working properly and PSG sounded weird after the
  production runtime install.
- Root cause: production LWJGL playback had been moved onto the
  stream-backed runtime FIFO. That runtime produces frame-sized slices, while
  OpenAL queues fixed-size buffers and expects enough forward PCM to keep its
  queue fed. The mismatch could put silence/partial buffers into normal
  forward playback and change timing-sensitive fade/PSG presentation.
- Fix: production LWJGL forward playback now uses the legacy stream path again
  for normal music/SFX, preserving existing fade and PSG behavior. LWJGL owns a
  bounded PCM history ring populated from the final mixed legacy output.
  During reverse presentation it drains that history through a reverse cursor.
  The stream-backed deterministic runtime remains tested and available for
  synthetic/runtime work, but LWJGL no longer advertises deterministic runtime
  presentation for ordinary forward output.

  LWJGL can switch its fill path to reverse-history drain without changing
  normal playback.
