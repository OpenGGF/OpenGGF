Temporary Rewind Support Blueprint
==================================

Purpose
-------

This temporary blueprint defines the high-level route to deterministic rewind
coverage for every Sonic 1, Sonic 2, and Sonic 3&K gameplay encounter. It is
not a final design doc. Use it as the shared reference for smaller design docs
covering field policy, helper snapshots, object references, child objects,
encounter validation, and torture-test enablement.

Definition Of 100% Gameplay Rewind Support
------------------------------------------

For a gameplay segment between explicit rewind boundaries, every runtime-owned
field that can affect future simulation must be one of:

1. Captured exactly and restored exactly.
2. Rebuilt deterministically from other restored state.
3. Rebound through a stable identity after restore.
4. Proven structural/non-gameplay and annotated as transient.
5. Explicitly deferred with a reason and a tracked follow-up.

The end state is not just "tests pass." The end state is that the unsupported
field inventory has no uncategorized gameplay state, all object families have
coverage, and rewind+replay produces the same simulation as uninterrupted play.

Scope And Boundaries
--------------------

In scope:
- Level gameplay for Sonic 1, Sonic 2, and Sonic 3&K.
- Player and sidekick state.
- Runtime-owned managers and registries.
- Level objects, badniks, bosses, projectiles, fragments, debris, particles,
  springs, platforms, solids, triggers, and zone-specific mechanics.
- Runtime palette, tile, layout mutation, scroll, render-mode, and special
  render state where it affects gameplay or replay determinism.

Out of scope unless a future design expands it:
- Menus, title screens, data select, and non-gameplay UI.
- Audio scrubbing parity.
- Editor undo/redo semantics.
- Rewind across hard level/mode transitions.

Hard rewind boundaries should clear or re-anchor history:
- level load
- act transition
- special stage entry/exit
- death/respawn until fully supported
- title/menu/data-select transitions
- editor entry/exit

Core Invariants
---------------

1. Rewind test data must be diagnostic input only. Do not hydrate engine state
   from trace data in committed tests.
2. Rewind validation compares engine uninterrupted play against engine
   rewind+replay. ROM trace replay remains a separate engine-vs-ROM invariant.
3. Raw Java object references are not durable rewind state unless the referenced
   object is guaranteed to survive restore in place.
4. Every new gameplay field in a runtime-owned class must have a rewind policy
   at introduction time.
5. Cosmetic-only state may be transient only when it cannot affect collision,
   object spawning, player control, timers, RNG, layout, or future gameplay.
6. The default path for object rewind support should be automatic state capture.
   Per-object custom snapshot code should be exceptional, not routine.
7. Automatic capture must be schema-driven and compact. It should not allocate
   reflection maps, boxed values, or per-field wrapper objects on every
   keyframe.

Automatic Stateful Capture Architecture
---------------------------------------

The preferred architecture is a high-level class-schema system:

- RewindSchemaRegistry: owns one cached RewindClassSchema per runtime class.
- RewindClassSchema: immutable field plan for a class hierarchy.
- RewindFieldPlan: precomputed access path, field policy, and compact codec.
- RewindCodec: reads/writes one supported data shape without per-frame policy
  decisions.
- RewindIdentityTable: converts object/player references to stable ids and
  resolves them after restore.
- RewindObjectStateBlob: compact stored state for one object instance.
- RewindStateBuffer: append-only primitive/object-id storage used by codecs.

The registry builds schemas once, either lazily on first capture or during a
startup audit. Per-frame capture then runs the already-compiled schema:

1. Look up the class schema.
2. Write primitive fields into primitive arrays or a byte buffer.
3. Write reference fields as stable integer ids.
4. Write supported helper state through nested codecs.
5. Write collection/map state only through approved specialized codecs.

Restore uses the same schema in reverse. Reference fields are restored in two
phases: write scalar/helper state first, then rebind object/player references
after managers have restored all possible targets.

This keeps the common case automatic while avoiding per-object snapshot classes.
Custom object snapshots remain available only for unusual cases where a class
has behavior that cannot be represented by field codecs.

Default Field Policy For Automatic Capture
------------------------------------------

Default behavior should minimize required annotations:

- static fields: ignored
- Java transient fields: ignored
- synthetic fields: ignored
- final primitive/String/enum/config fields: treated as structural by default
- mutable primitive/String/enum fields: captured
- arrays of supported values: captured with compact array codecs
- records containing immutable primitive/String/enum values: captured as opaque
  values in the first schema milestone; records with mutable components such as
  arrays or BitSet remain unsupported until flattened record codecs exist
- RewindStateful helpers: captured through their own compact state
- ObjectInstance/PlayableEntity/player refs: captured as stable ids
- unsupported collections/maps: rejected unless a specialized codec exists
- renderer/art/cache/config objects: structural by inferred type where safe

Annotations should force behavior, not carry the normal burden:

- @RewindCapture: force capture for a field that default policy would skip.
- @RewindTransient: force skip with a required reason.
- @RewindDeferred: known gameplay gap with a required reason.
- @RewindReference: force a specific rebinding policy.
- @RewindStructural: mark constructor/ROM/config state explicitly when inference
  is not obvious.

The goal is that ordinary object authors can add fields such as ints, booleans,
small enums, helper state, and player/object refs without writing per-object
rewind code. The schema audit should explain unsupported fields in terms of the
missing codec or ambiguous policy.

Annotation cleanup is part of the work, not a cosmetic follow-up. Whenever a
new default policy or codec makes a previously annotated field inferable, remove
the now-redundant annotation unless it documents a real override. The steady
state should be low-noise: annotations identify exceptional behavior, deferred
gaps, or non-obvious structural state.

Annotation Minimization Strategy
--------------------------------

Annotation minimization is a design constraint, not a cleanup preference. The
rewind rollout must not fix inventory failures by spraying field annotations
across object classes. Field annotations should remain exceptional and should
mean that the inferred policy would be ambiguous or wrong.

Prefer these mechanisms, in order, before adding per-field annotations:

1. Default scalar/value policy for ordinary mutable primitives, Strings, enums,
   small immutable records, and arrays of supported values.
2. Central type policy for repeated helper, runtime, renderer, art, config,
   service, manager, player, and object-reference types.
3. Package or hierarchy policy where ownership boundaries are clear enough to
   infer structural/runtime state safely.
4. Shared codecs for repeated helper and collection shapes.
5. Base-class or class-family rules for common object and sprite patterns.
6. Field annotations only for overrides, known deferred gaps, or structural
   state that inference cannot prove.

If more than a small number of fields need the same annotation pattern, treat
that as a missing type policy, codec, base-class rule, or package rule. Add the
shared rule first, then rerun the inventory before annotating individual fields.

The inventory tool should report annotation density by class, declared type, and
package. It should also identify annotations that are now redundant because the
schema would infer the same policy. Redundant annotations should warn at first
and eventually fail in generic-eligible classes once the relevant policy is
stable.

Memory And Performance Model
----------------------------

Automatic capture is only acceptable if it stays compact and fast.

Memory rules:

- Store field values in dense primitive arrays or byte buffers, not maps.
- Store references as integer ids, not object references.
- Store enum values as ordinals or stable small ids.
- Store booleans in packed bitsets where field counts justify it.
- Deduplicate immutable schema metadata across all instances of a class.
- Avoid storing structural finals and ROM/config/art/cache fields.
- Store collections/maps only through compact specialized codecs.
- Prefer fixed-layout blobs for common classes.

Performance rules:

- Build field plans once per class; never rediscover fields during gameplay.
- Use MethodHandle/VarHandle or equivalent cached accessors instead of raw
  reflective lookup on every capture.
- Avoid boxing primitives during capture/restore.
- Avoid per-field allocation during keyframe capture.
- Separate capture buffers by primitive kind when that improves locality.
- Keep custom capture paths rare and measurable.
- Add benchmark counters for bytes per object, capture time, restore time, and
  schema cache hit rate.

A schema-backed generic capture path should be fast enough to run for every
live object on keyframe capture. Held rewind remains performant by seeking to a
keyframe and replaying inputs forward, not by storing every frame as a full
snapshot.

Field Policy
------------

Every unsupported field must receive one of these decisions:

- captured: included in a generic or explicit snapshot.
- structural: ROM/config/art-derived or constructor-fixed; annotate
  @RewindTransient with a clear reason.
- deferred: known gameplay gap; annotate @RewindDeferred with a clear reason
  and track a follow-up.
- rebuilt: deterministically reconstructed during restore from captured parent
  state.
- rebound: restored through a stable identity rather than as a raw reference.

The full inventory tool should eventually report no unsupported fields except
intentional deferred work. The enabled audit should remain stricter for classes
already marked generic-eligible.

Reusable Snapshot Support
-------------------------

Do not solve repeated helper types independently in each object. Add shared
support for common state containers first:

- SubpixelMotion.State
- ObjectAnimationState
- PlatformBobHelper
- SolidObjectParams when mutable or behavior-affecting
- AnimationTimer
- boss explosion controllers
- per-player/player-index state records
- small immutable records and arrays used by object helpers

Preferred patterns:
- Implement RewindStateful on reusable helper types when they own behavior
  state and can restore themselves cleanly.
- Use small snapshot records when helper internals should stay encapsulated.
- Treat structural renderer/art/config helpers as transient only after proving
  they do not mutate gameplay decisions.

Stable Reference Rebinding
--------------------------

Object and player references must be restored by stable identity. Design docs
for reference-heavy objects should use these identity classes:

- main player
- sidekick by sidekick index
- object by runtime slot id plus generation
- object by persistent spawn id where slot identity is not stable enough
- child object by parent id plus child role/index

Fields requiring this policy include:
- AbstractPlayableSprite
- PlayableEntity
- ObjectInstance
- PowerUpObject
- InstaShieldHandle
- locked/grabbed/captured player fields
- parent/child object refs
- boss component refs

Restore ordering:
1. Restore managers and object/sprite containers.
2. Recreate or restore object instances.
3. Restore scalar/helper state.
4. Rebind references after all candidate targets exist.
5. Run a post-restore validation pass that fails fast on missing required refs.

Child Object And Spawn Graph Policy
-----------------------------------

Every object that creates children needs an explicit graph policy:

- parent-owned children: parent snapshots child list and child state.
- independent children: child instances are normal rewindable objects with
  stable runtime identity.
- deterministic children: parent captures enough state to respawn them exactly.
- cosmetic children: annotated transient only if they cannot affect gameplay.

This applies to debris, fragments, projectiles, boss parts, bridge sections,
animals, splash/dust, starpost effects, bubbles, signpost effects, and particle
objects that can interact with players or other objects.

Object Coverage Contract
------------------------

Every concrete object/badnik/boss class must satisfy exactly one coverage path:

1. Generic capture eligible and passes field audit.
2. Explicit capture/restore implementation with tests.
3. Stateless/structural class with all mutable fields categorized.
4. Deferred class with reason and known gameplay limitation.

New object work must not introduce uncategorized mutable gameplay fields.

Validation Strategy
-------------------

Use multiple validation layers. Each layer catches a different failure mode.

1. Inventory validation:
   - unsupported field inventory trends toward zero
   - enabled audit catches regressions in generic-eligible classes
   - transient/deferred annotations require reasons

2. Unit validation:
   - helper snapshot round trips
   - reference rebinding tests
   - child graph restore tests
   - manager/registry snapshot tests

3. Encounter validation:
   - uninterrupted run compared with rewind+replay run
   - scenario starts near an active interaction
   - seek backward, replay with identical input, compare state hashes
   - catalog organized by game, zone, object family, and special mechanic

4. Trace replay validation:
   - separate ROM-vs-engine parity tests remain diagnostic
   - trace replay failures are not fixed by syncing engine state from traces
   - rewind tests may reuse trace inputs but compare engine paths against each
     other

5. Torture validation:
   - adjacent fixed rewinds
   - long progressive rewinds
   - randomized seek patterns
   - object-heavy zones
   - sidekick and multi-sidekick cases

Encounter Catalog Shape
-----------------------

Area-specific design docs should add rows to a catalog with this shape:

- game
- zone/act
- object/mechanic family
- setup method
- required input sequence
- checkpoint frame or state predicate
- rewind distance
- comparison fields
- known limitations

Representative categories:
- springs and forced movement
- monitors and power-ups
- spikes/hazards/hurt flow
- solids/platforms/riding/carrying
- destructible blocks and fragments
- object projectiles
- boss components and defeat flows
- water/bubbles/drowning
- palette/tile/layout mutation triggers
- sidekick interactions and respawn behavior

Implementation Priority
-----------------------

Recommended order:

1. Schema registry and compact storage foundation.
   Build RewindSchemaRegistry, RewindClassSchema, RewindFieldPlan, codecs, and
   dense state buffers so the default path is automatic and memory-efficient.

2. Central type and package policy registry.
   Add reusable policy registration for common captured helper types, structural
   runtime/render/art/config/service types, standard reference types, and safe
   package or hierarchy-level defaults. This should be available before the
   object rollout starts so repeated field shapes are solved once.

3. Default automatic field policy.
   Capture mutable primitive/String/enum/record/value fields automatically,
   infer structural finals where safe, and reserve annotations for overrides.

4. Stable reference identity and rebinding.
   Add standard ids for players, sidekicks, object slots/generations, persistent
   spawns, and parent/child roles before tackling reference-heavy objects.

5. Shared helper codecs.
   Add compact support for SubpixelMotion.State, ObjectAnimationState,
   AnimationTimer, PlatformBobHelper, boss explosion controllers, and similar
   repeated helper state.

6. Collection and child-graph codecs.
   Support approved List/Map/Set shapes, parent-owned child lists, independent
   child objects, and deterministic respawn policies.

7. Annotation-density audit and default-policy cleanup.
   Track annotation counts by class, declared type, and package. Before adding
   repeated annotations, prefer a missing policy or codec. Remove
   @RewindTransient/@RewindDeferred annotations that are no longer needed
   because the schema can infer capture/structural behavior safely. Keep
   annotations only for overrides, deferred gaps, and non-obvious structural
   state.

8. Inventory-driven object rollout.
   Apply the automatic schema path across S3K first, then S2, then S1, using
   explicit object snapshots only for exceptional behavior the schema cannot
   represent. Rollout reviews should reject annotation-heavy fixes when a shared
   policy or codec would cover the same pattern.

9. Encounter catalog validation.
   Add engine uninterrupted vs rewind+replay scenarios for representative
   mechanics across all three games.

10. Torture test enablement.
   Enable adjacent, progressive, random, object-heavy, and sidekick torture
   tests incrementally once the catalog is stable.

Post-Audio-Rewind Follow-Up
---------------------------

After audio rewind is complete, add rewind coverage for entering/exiting level
debug mode and movement performed while debug mode is active. Keep the main
task brief here; detailed notes live in:

  DEBUG_MODE_REWIND_FOLLOWUP.tmp.txt

Success Criteria
----------------

The overall rewind effort is complete when:

1. The full unsupported inventory contains no uncategorized gameplay state.
2. All deferred entries are documented limitations with explicit owners.
3. Every concrete object class has a coverage path.
4. Encounter replay passes across representative S1/S2/S3K mechanics.
5. Rewind torture tests are enabled and stable for long randomized sessions.
6. Full rewind suite stays green without manual inventory generators appearing
   as skipped tests.
