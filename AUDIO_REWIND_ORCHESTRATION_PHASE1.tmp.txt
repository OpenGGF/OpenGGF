Audio Rewind Orchestration - Phase 1 Synthesis
==============================================

Purpose
-------

This temporary note captures the first multi-agent exploration pass for audio
rewind. It is an orchestration artifact, not an implementation plan. The goal is
to convert the blueprint into coordinated work tracks with clear first
milestones, ownership boundaries, and unresolved design questions.


Phase 1 Status
--------------

Completed read-only exploration tracks:

1. Hardware timing and accuracy model.
2. Current audio backend ownership and presentation seams.
3. Rewind, pause, frame-step, and SegmentCache integration.
4. Test and benchmark strategy.

No agents edited source files.


Shared Findings
---------------

1. Current audio advancement is host-buffer-driven.

   LWJGLAudioBackend.fillBuffer() calls into AudioStream.read(), which advances
   SMPS/chip state while satisfying OpenAL buffer demand. Starting a stream also
   prefills three 1024-frame buffers, so issuing a play command can advance
   authoritative audio before the gameplay timeline advances another frame.

2. AudioManager is the practical command boundary.

   Gameplay-facing calls such as playMusic, playSfx, donor routing, fade, stop,
   tempo, and speed-shoes operations already funnel through AudioManager in most
   cases. This is the lowest-risk Tier 0 suppression point and the likely Tier 1
   command timeline recorder.

3. AudioBackend mixes authoritative state and presentation state.

   LWJGLAudioBackend owns OpenAL device/source/buffer objects and scratch buffers,
   but it also owns logical state such as currentMusicId, musicStack,
   pendingRestore, sfxBlocked, speedShoesEnabled, current streams, and SMPS
   driver instances. That split needs to be made explicit before deterministic
   rewind can be correct.

4. SegmentCache must not store heavy audio snapshots.

   Held rewind expands per-frame CompositeSnapshot strips. Registering full
   SMPS/chip/backend state in the normal rewind registry would multiply audio
   snapshot cost across every cached frame. Audio needs its own sparse keyframe
   store plus command replay.

5. Pause/frame-step currently advances audio incidentally.

   Paused idle returns before audio update. Paused frame-step currently falls
   through to AudioManager.update(), but that means "poll/fill OpenAL once", not
   "advance one deterministic audio frame". The architecture should keep the UI
   behavior but replace the implementation with an explicit one-frame audio
   slice when the deterministic core exists.


Coordinator Conclusion
----------------------

The first implementation milestone should be Tier 0 suppression plus an explicit
frame-step seam, not immediate SMPS/chip snapshots.

Reasoning:

1. Tier 0 removes the most visible rewind defect: internal seek/segment replay
   can currently emit duplicate SFX, restart music, or mutate live OpenAL state.
2. It is small enough to test with fake backends and the existing rewind
   controller.
3. It avoids pretending the current buffer-driven SMPS advancement is
   deterministic.
4. It creates the scope guard and lifecycle hooks needed by every later tier.
5. It lets command timeline and deterministic audio-core design proceed in
   parallel without blocking the first user-visible improvement.


Recommended First Milestone
---------------------------

Name:

Tier 0 Audio Rewind Suppression And Frame-Step Seam

Goal:

During internal rewind replay and segment expansion, gameplay may execute audio
commands, but those commands must not dispatch to the live backend or enqueue
audible presentation side effects. Paused frame-step should also gain an explicit
audio-frame hook, even if the hook initially delegates to existing behavior.

Included:

1. Add a central replay/suppression scope opened by rewind seek and held-rewind
   segment expansion.
2. Enforce suppression at AudioManager command entry points.
3. Add post-restore/release hooks so presentation can be flushed, stopped, or
   restarted according to a policy instead of leaking replay artifacts.
4. Add explicit frame-step audio hook in the GameLoop path where paused
   frame-step currently reaches AudioManager.update().
5. Add fake-backend tests proving internal rewind replay does not increase live
   backend call counts.

Excluded from this milestone:

1. Command timeline replay.
2. Deterministic frame-owned SMPS/chip advancement.
3. SMPS driver or sequencer snapshots.
4. YM2612, PSG, DAC, or resampler snapshots.
5. Audible reverse scrubbing.
6. Sample parity guarantees.


Initial API Shape To Refine
---------------------------

The agents converged on this shape as a useful starting point:

    interface AudioRewindCoordinator {
        RewindReplayScope beginInternalReplay(int fromFrame,
                                              int targetFrame,
                                              RewindReplayReason reason);
        void afterRestore(int frame, AudioPresentationPolicy policy);
        void discardAfter(int frame);
        void resetAtFrame(int frame);

        void beginGameplayAudioFrame(int frame, AudioFrameMode mode);
        void recordCommand(AudioCommand command);
        void advanceAuthoritativeFrame(AudioAdvancePolicy policy);
        void endGameplayAudioFrame();
    }

For Tier 0, only a subset is needed:

1. beginInternalReplay(...), returning an AutoCloseable scope.
2. afterRestore(...), for release/presentation cleanup.
3. beginGameplayAudioFrame(...) or a narrower frame-step hook.

The command timeline and deterministic advancement methods should be designed
now, but can remain unimplemented until Tier 1 and Tier 2.

Open Questions
--------------

1. Should suppressed internal replay commands be discarded entirely in Tier 0, or
   counted diagnostically for tests and future timeline validation?
2. What is the default post-restore policy after held rewind release: restart
   current music, silence transient SFX, flush all presentation, or defer until
   Tier 1 command replay can reconstruct exact intent?
3. How should SmpsSequencer-originated callbacks such as restoreMusic during
   fade-in be classified: gameplay command, internal driver event, or backend
   intent?
4. Which direct AudioBackend accessors are legitimate debug/presentation controls
   and which must be migrated behind AudioManager before Tier 1?
5. Should AudioManager remain global with a gameplay-scoped coordinator attached,
   or should audio rewind state live entirely inside GameplayModeContext and be
   passed into AudioManager as an optional policy?
6. What fixed output-rate policy should deterministic tests use before the
   backend/core split is complete?


Phase 2 Synthesis
=================

Phase 2 completed the design and review tracks. The recommendation still starts
with Tier 0, but the self-review tightened the milestone:

Tier 0 is not merely "skip backend calls while rewinding." It must suppress
internal replay without mutating durable audio intent, and it must classify or
close direct backend bypasses that would evade AudioManager.


Revised First Milestone
-----------------------

Name:

Tier 0 Audio Rewind Suppression, Bypass Guardrails, And Frame-Step Seam

Goal:

Internal rewind restore/replay paths must not emit live audio presentation
effects or mutate durable audio intent such as ring alternation, speed state,
fade/restore intent, or music override state. Paused frame-step gets an explicit
audio hook, but Tier 0 does not claim deterministic audio advancement yet.

Claims allowed after Tier 0:

1. Internal rewind replay does not duplicate live SFX/music commands through the
   guarded command path.
2. Rewind replay scopes are nestable, exception-safe, and cover restore plus
   forward replay.
3. Frame-step has an explicit audio seam for later deterministic advancement.
4. Presentation cleanup after rewind release is conservative and does not claim
   exact music phase restoration.

Claims not allowed after Tier 0:

1. Exact audio phase restoration.
2. Exact intended music restoration from a historical frame.
3. Deterministic paused frame-step audio.
4. Command timeline replay.
5. SMPS/chip/sample parity.


Tier 0 API Recommendation
-------------------------

Add small replay/presentation types near the audio rewind integration:

    enum AudioReplayReason {
        SEEK,
        STEP_BACKWARD,
        SEGMENT_EXPANSION
    }

    enum AudioPresentationPolicy {
        SUPPRESSED_INTERNAL_RESTORE,
        STOP_TRANSIENT_SFX_RESYNC_MUSIC,
        STOP_ALL_PRESENTATION
    }

    interface AudioReplayScope extends AutoCloseable {
        @Override
        void close();
    }

Expose a narrow policy surface through AudioManager or a gameplay-scoped
coordinator attached to AudioManager:

    AudioReplayScope beginRewindReplay(int fromFrame,
                                       int targetFrame,
                                       AudioReplayReason reason);
    boolean isRewindReplaySuppressed();
    void afterRewindRestore(int frame, AudioPresentationPolicy policy);
    void advancePausedFrameStepAudio();

Implementation rules:

1. Use a depth counter; nested scopes are valid.
2. Closing a scope is idempotent and cannot underflow.
3. The guard must run before command normalization that mutates durable intent.
4. Setup/lifecycle methods remain allowed: backend/profile/ROM/sound-map/donor
   registration/reset/destroy.
5. `advancePausedFrameStepAudio()` may delegate to legacy `update()` in Tier 0,
   but its documentation must say it is not deterministic until the later
   authoritative audio-core split.


Tier 0 Call-Site Plan
---------------------

Wrap all restore/replay paths:

1. RewindController.seekTo(...): scope covers keyframe restore, stepper priming,
   every forward replay step, final restore state, and internal cleanup.
2. RewindController.stepBackward(...): scope covers SegmentCache lookup/cold
   expansion, restoreKeyframe, stepAndCapture loop, final registry restore, and
   stepper priming.
3. SegmentCache.snapshotAt(...): optional nested scope around cold expansion if
   direct use outside RewindController remains possible.

Guard AudioManager commands before durable state mutation:

1. playMusic, including system-command speed-shoes handling.
2. all playSfx overloads, especially GameSound.RING before ringLeft toggles.
3. playDonorSfx and playDonorMusic.
4. endMusicOverride, changeMusicTempo, stopAllSfx, stopMusic.
5. fadeOutMusic overloads.

Add release/presentation hooks:

1. LiveRewindManager release path: STOP_TRANSIENT_SFX_RESYNC_MUSIC.
2. LiveRewindManager.clear while rewinding: stronger cleanup.
3. TraceSessionLauncher trace rewind release: STOP_TRANSIENT_SFX_RESYNC_MUSIC.
4. TraceSessionLauncher teardown: STOP_ALL_PRESENTATION.

Add GameLoop frame-step seam:

    if (doFrameStep) {
        audioManager.advancePausedFrameStepAudio();
    } else {
        audioManager.update();
    }


Bypass Guardrails
-----------------

Before Tier 1, these gameplay-relevant direct backend calls must either migrate
behind AudioManager or be explicitly classified out of gameplay rewind scope:

1. SmpsSequencer restoreMusic callback.
2. Sonic3kCoordFlagHandler restoreMusic callback.
3. AizMinibossCutsceneInstance restoreMusic callback.
4. LevelManager setSpeedShoes(false).
5. Sonic3kSpecialStageManager setSpeedMultiplier(...) calls.

Guardrail recommendation:

1. Add AudioManager methods for restoreMusic, setSpeedShoes, and
   setSpeedMultiplier so gameplay code no longer needs getBackend().
2. Treat AudioManager.getBackend() as a controlled escape hatch for debug,
   tests, and presentation-only controls.
3. Add a scanner test before Tier 1 that fails on gameplay getBackend() bypasses
   except for an explicit allowlist.


Tier 1 Command Timeline Design
------------------------------

Timeline records should be normalized after AudioManager resolves donor, ring,
system-command, and fallback routing. Raw request metadata may be retained for
diagnostics, but replay should use resolved commands.

Model sketch:

    record AudioTimelineEntry(long frame, int order, AudioCommand command) {}

    sealed interface AudioCommand {
        record PlayMusic(int musicId, MusicRoute route, boolean override,
                         String donorGameId) implements AudioCommand {}
        record PlaySfx(int sfxId, String sfxName, SfxRoute route, float pitch,
                       String donorGameId) implements AudioCommand {}
        record FadeOutMusic(int steps, int delay) implements AudioCommand {}
        record StopMusic() implements AudioCommand {}
        record StopAllSfx() implements AudioCommand {}
        record EndMusicOverride(int musicId) implements AudioCommand {}
        record RestoreMusic(RestoreCause cause) implements AudioCommand {}
        record SetSpeedShoes(boolean enabled) implements AudioCommand {}
        record SetSpeedMultiplier(int multiplier) implements AudioCommand {}
        record ChangeMusicTempo(int dividingTiming) implements AudioCommand {}
        record ResetRingAlternation(boolean ringLeft) implements AudioCommand {}
    }

Key rules:

1. Frame comes from the gameplay/audio coordinator.
2. Order is a per-frame monotonic sequence assigned before backend dispatch.
3. Ring commands record the resolved RING_LEFT/RING_RIGHT result.
4. Speed shoes are semantic commands, not opaque music ids.
5. discardAfter(frame) removes abandoned future commands after live rewind.


Tier 2 Deterministic Core Design
--------------------------------

Target direction:

    GameLoop frame N
      -> AudioManager records/dispatches commands
      -> AudioRuntime.advanceFrame(N)
           renders a fixed sample slice into AudioPcmFifo
      -> LWJGLAudioBackend.update()
           drains AudioPcmFifo into OpenAL only

Core pieces:

1. AudioRenderClock with outputSampleRate, regionFrameRate, and fixed-point
   fractional accumulator.
2. AudioPcmFifo as presentation staging, not rewind authority.
3. LWJGLAudioBackend.fillBuffer() drains FIFO in core mode, falls back to legacy
   AudioStream.read(...) only while migrating.
4. startStream() must eventually stop pre-rendering authoritative audio in core
   mode.
5. Deterministic tests should start with fixed 44100 Hz, matching existing audio
   regression expectations.

Tier 2 migration can be incremental:

1. Add no-op advanceAuthoritativeFrame() seam that delegates to legacy update.
2. Add FIFO and drain path with legacy fallback.
3. Move SMPS stream ownership into a core above OpenAL.
4. Make OpenAL presentation-only.
5. Add sparse audio keyframe restore/replay.


Snapshot Boundary
-----------------

CompositeSnapshot should carry at most a small audio cursor. Heavy audio state
belongs in sparse audio keyframes outside SegmentCache.

Tier 1 intent/keyframe state:

1. Audio profile/loader/donor identities.
2. ringLeft and command timeline cursor.
3. currentMusicId, override stack ids, pendingRestore, sfxBlocked,
   speedShoesEnabled, speed multiplier, tempo intent.

Tier 2 SMPS logical state:

1. backend logical stream graph.
2. SmpsDriver sequencer ordering, SFX set, locks, latches, continuous SFX state.
3. SmpsSequencer scalar state, tempo/sample counters, speed/fade state.
4. track interpreter state, loop/return stacks, modulation, envelopes, voice and
   frequency state.

Tier 3 chip/sample state:

1. VirtualSynthesizer output rate.
2. YM2612 DAC, timers, LFO, EG, channel/operator, feedback, resampler state.
3. PSG registers, counters, latch, LFSR, clock fraction, blip state.
4. BlipResampler/BlipDeltaBuffer internal history.

Never snapshot:

1. OpenAL device/context/source/buffer ids.
2. queued/processed OpenAL buffer counts.
3. native WAV buffers and transient SFX sources.
4. scratch arrays/direct buffers.
5. debug UI mute/solo by default.

1. TestAudioManagerRewindSuppression.
2. TestRewindControllerAudioSuppression.
3. TestSegmentCacheAudioSuppression only if SegmentCache gets its own guard.

Important Tier 0 assertions:
